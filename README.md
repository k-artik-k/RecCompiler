# RecCompiler

### A Mini C Compiler with Full Recursion Support

RecCompiler is a **self-contained mini C compiler** written entirely in C. It compiles a meaningful subset of the C language down to bytecode and executes it on a built-in **stack-based virtual machine**. The compiler handles **recursive functions** (including mutual/indirect recursion) through proper stack frame management.

No external assembler or linker required — everything runs inside one binary.

✨ **Version 2.0 update: RecCompiler now features two versions!** ✨
1. **The GUI Electron Desktop App (NEW):** A gorgeous, interactive application built with Electron, offering real-time visualization of the complete compilation pipeline (Tokens -> AST -> Bytecode -> Execution) with a sleek code editor.
2. **The Classic Terminal Version:** A lightweight, pure C compiler built as a single `.exe` file.

## Compiler Architecture Overview

The compiler processes source code through **four distinct phases**, each transforming the program into a lower-level representation:

```
┌─────────────┐     ┌──────────┐     ┌─────────┐     ┌──────────────┐     ┌────────┐
│ Source Code  │────>│  LEXER   │────>│ PARSER  │────>│   COMPILER   │────>│   VM   │
│   (.c file)  │     │ (tokens) │     │  (AST)  │     │  (bytecode)  │     │ (exec) │
└─────────────┘     └──────────┘     └─────────┘     └──────────────┘     └────────┘
   Characters     Token Stream      Syntax Tree      Bytecode Array       Output
```

Each phase is implemented in its own module:

| Phase | Module | Input | Output |
|-------|--------|-------|--------|
| **1. Lexer** | `lexer.c` | Raw characters | Token stream |
| **2. Parser** | `parser.c` | Token stream | Abstract Syntax Tree (AST) |
| **3. Compiler** | `compiler.c` | AST | Bytecode (int array) |
| **4. VM** | `vm.c` | Bytecode | Program output |

We will trace through each phase using this example program:

```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}

int main() {
    int x = 5;
    print(factorial(x));
    return 0;
}
```

## Phase 1 — Lexical Analysis (Lexer)

**File:** `src/lexer.c` · `src/lexer.h`

### What It Does

The lexer (also called a **scanner** or **tokeniser**) reads the raw source code **character by character** and groups them into meaningful units called **tokens**. It strips out whitespace and comments, and identifies keywords, identifiers, numbers, and operators.

Think of it as converting a sentence into individual words with labels.

### How It Works

The lexer maintains a pointer into the source string. On each call to `lexer_advance()`, it:

1. **Skips whitespace and comments** (spaces, tabs, newlines, `//...`, `/*...*/`)
2. **Reads the next character** and decides what kind of token it starts
3. **Builds the full token**:
   - If it's a digit → read all consecutive digits to form a number
   - If it's a letter or `_` → read the full identifier, then check against keywords
   - If it's an operator → check for two-character operators (`==`, `!=`, `<=`, `>=`, `&&`, `||`)
4. **Returns a `Token` struct** with the type, text value, and line number

### Token Types

```
Keywords:     TOKEN_INT  TOKEN_VOID  TOKEN_IF  TOKEN_ELSE
              TOKEN_WHILE  TOKEN_FOR  TOKEN_RETURN

Literals:     TOKEN_NUMBER  (integer constant like 42)
              TOKEN_IDENT   (identifier like "factorial" or "n")

Operators:    TOKEN_PLUS(+)  TOKEN_MINUS(-)  TOKEN_STAR(*)
              TOKEN_SLASH(/)  TOKEN_PERCENT(%)
              TOKEN_EQ(==)  TOKEN_NEQ(!=)  TOKEN_LT(<)  TOKEN_GT(>)
              TOKEN_LTE(<=)  TOKEN_GTE(>=)
              TOKEN_AND(&&)  TOKEN_OR(||)  TOKEN_NOT(!)
              TOKEN_ASSIGN(=)

Punctuation:  TOKEN_LPAREN(()  TOKEN_RPAREN())
              TOKEN_LBRACE({)  TOKEN_RBRACE(})
              TOKEN_SEMI(;)  TOKEN_COMMA(,)

Special:      TOKEN_EOF  TOKEN_ERROR
```

### Example: Lexer Output

For our example program, the lexer produces this token stream:

```

```

### Key Design Decisions

- **Hand-written scanner** (not generated by a tool like Lex/Flex) — simpler, easier to debug, and educational
- **Keyword detection**: identifiers are scanned first, then checked against a keyword table. If `"int"` matches a keyword, it becomes `TOKEN_INT` instead of `TOKEN_IDENT`
- **Lookahead**: for two-character operators like `<=`, the lexer peeks at the next character before deciding the token type
- **Line tracking**: each token records its source line number for error messages

## Phase 2 — Parsing (Parser)

**File:** `src/parser.c` · `src/parser.h` · `src/ast.h` · `src/ast.c`

### What It Does

The parser takes the flat stream of tokens and builds a hierarchical **Abstract Syntax Tree (AST)**. The AST represents the grammatical structure of the program — which expressions are nested inside which statements, which statements belong to which functions, etc.

### Parsing Technique: Recursive Descent

RecCompiler uses a **recursive-descent parser**. This is one of the most intuitive parsing techniques:

- Each grammar rule (like "an if-statement is `if` `(` expression `)` statement") becomes a **C function** that consumes the corresponding tokens
- These functions call each other recursively, mirroring the grammar's structure
- The parser is hand-written, not generated

### The Grammar

The parser implements this grammar (in EBNF notation):

```
program      → func_def*
func_def     → type IDENT '(' params? ')' block
params       → type IDENT (',' type IDENT)*
type         → 'int' | 'void'

block        → '{' statement* '}'
statement    → var_decl | if_stmt | while_stmt | for_stmt
             | return_stmt | expr_stmt | block

var_decl     → type IDENT ('=' expr)? ';'
if_stmt      → 'if' '(' expr ')' statement ('else' statement)?
while_stmt   → 'while' '(' expr ')' statement
for_stmt     → 'for' '(' init? ';' expr? ';' expr? ')' statement
return_stmt  → 'return' expr? ';'
expr_stmt    → expr ';'

expr         → assignment
assignment   → IDENT '=' assignment | logic_or
logic_or     → logic_and ('||' logic_and)*
logic_and    → equality ('&&' equality)*
equality     → comparison (('==' | '!=') comparison)*
comparison   → addition (('<' | '>' | '<=' | '>=') addition)*
addition     → multiply (('+' | '-') multiply)*
multiply     → unary (('*' | '/' | '%') unary)*
unary        → ('-' | '!') unary | postfix
postfix      → primary ('(' arguments? ')')?
primary      → NUMBER | IDENT | '(' expr ')'
```

### Operator Precedence

The grammar encodes **operator precedence** through the nesting of rules. Operators at the bottom of the grammar tree bind tighter:

```
Lowest precedence:   =          (assignment, right-assoc)
                     ||         (logical OR)
                     &&         (logical AND)
                     == !=      (equality)
                     < > <= >=  (comparison)
                     + -        (addition)
                     * / %      (multiplication)
Highest precedence:  - !        (unary)
                     f()        (function call)
                     ()         (grouping)
```

For example, `n * factorial(n - 1)` is parsed as:

```
    BINARY(*)
    ├── IDENT(n)
    └── CALL(factorial)
        └── BINARY(-)
            ├── IDENT(n)
            └── NUMBER(1)
```

Multiplication has lower precedence than function call, so `factorial(n-1)` is evaluated first, then multiplied by `n`.

### AST Node Types

```
NODE_PROGRAM      — top-level: contains function definitions
NODE_FUNC_DEF     — function: name, return type, params, body
NODE_BLOCK        — { statements... }
NODE_VAR_DECL     — int x = expr;
NODE_IF           — if (cond) body [else else_body]
NODE_WHILE        — while (cond) body
NODE_FOR          — for (init; cond; update) body
NODE_RETURN       — return expr;
NODE_EXPR_STMT    — expr;  (expression used as statement)
NODE_ASSIGN       — x = expr
NODE_BINARY       — expr OP expr
NODE_UNARY        — OP expr
NODE_CALL         — func(args...)
NODE_NUMBER       — integer literal
NODE_IDENT        — variable reference
```

### Example: AST Output

For our factorial program, the parser produces this tree:

```
PROGRAM
├── FUNC_DEF "factorial" returns:int params:[n:int]
│   └── BLOCK
│       ├── IF
│       │   ├── cond: BINARY(<=)
│       │   │         ├── IDENT "n"
│       │   │         └── NUMBER 1
│       │   └── body: RETURN
│       │             └── NUMBER 1
│       └── RETURN
│           └── BINARY(*)
│               ├── IDENT "n"
│               └── CALL "factorial"
│                   └── arg0: BINARY(-)
│                             ├── IDENT "n"
│                             └── NUMBER 1
│
└── FUNC_DEF "main" returns:int params:[]
    └── BLOCK
        ├── VAR_DECL "x" type:int
        │   └── init: NUMBER 5
        ├── EXPR_STMT
        │   └── CALL "print"
        │       └── arg0: CALL "factorial"
        │                 └── arg0: IDENT "x"
        └── RETURN
            └── NUMBER 0
```

### Key Design Decisions

- **Recursive descent** is chosen because it naturally maps to the grammar and is easy to understand and extend
- **No separate semantic analysis pass** — type checking and scope analysis are done during compilation (Phase 3) for simplicity
- **AST uses a single flexible node struct** with multiple optional fields, rather than separate structs for each node type. This trades memory efficiency for code simplicity

## Phase 3 — Bytecode Compilation (Compiler)

**File:** `src/compiler.c` · `src/compiler.h`

### What It Does

The compiler walks the AST and emits a flat array of **bytecode instructions** for the virtual machine. It also manages:

- A **symbol table** for tracking local variables and their stack slots
- A **function table** for tracking function addresses in bytecode
- **Forward reference patching** for functions called before they're defined

### The Bytecode Instruction Set

RecCompiler's VM uses a **stack-based architecture** (like Java's JVM or Python's bytecode). All operations work with a value stack — no registers.

```
Instruction          Operands        Description
─────────────────────────────────────────────────────────
OP_CONST <value>     1 (int)         Push constant onto stack
OP_POP               0               Discard top of stack

OP_ADD               0               Pop 2, push a + b
OP_SUB               0               Pop 2, push a - b
OP_MUL               0               Pop 2, push a * b
OP_DIV               0               Pop 2, push a / b
OP_MOD               0               Pop 2, push a % b
OP_NEG               0               Negate top of stack

OP_EQ                0               Pop 2, push (a == b)
OP_NEQ               0               Pop 2, push (a != b)
OP_LT                0               Pop 2, push (a < b)
OP_GT                0               Pop 2, push (a > b)
OP_LTE               0               Pop 2, push (a <= b)
OP_GTE               0               Pop 2, push (a >= b)
OP_AND               0               Pop 2, push (a && b)
OP_OR                0               Pop 2, push (a || b)
OP_NOT               0               Logical not of top

OP_LOAD <slot>       1 (int)         Push stack[fp + slot]
OP_STORE <slot>      1 (int)         Pop → stack[fp + slot]

OP_JMP <addr>        1 (int)         Jump to address
OP_JZ <addr>         1 (int)         Pop; jump if zero

OP_CALL <addr> <n>   2 (addr, nargs) Call function
OP_RET               0               Return from function
OP_ENTER <n>         1 (int)         Allocate n local slots

OP_PRINT             0               Pop, print, push 0
OP_SCAN              0               Read int, push it
OP_HALT              0               Stop execution
```

### Compilation Strategy

#### Variables

Each variable (parameter or local) is assigned a **slot number** relative to the frame pointer (fp):

```
Parameters:  slot 0, 1, 2, ...  (in order of declaration)
Locals:      slot N, N+1, ...   (where N = number of params)
```

For `int factorial(int n)`:
- `n` is parameter 0 → slot 0

For `int main()` with `int x = 5;`:
- `x` is local 0 → slot 0  (no params, so locals start at 0)

#### Control Flow

Control flow is compiled using **jumps**:

```c
// Source:
if (n <= 1) return 1;

// Compiled to:
  LOAD 0          // push n
  CONST 1         // push 1
  LTE             // n <= 1 → 0 or 1
  JZ skip         // if false, jump over the return
  CONST 1         // push return value 1
  RET             // return
skip:
  ...             // continue with rest of function
```

#### Function Calls

```c
// Source:
factorial(n - 1)

// Compiled to:
  LOAD 0          // push n (argument value computation)
  CONST 1         // push 1
  SUB             // n - 1
  CALL addr 1     // call factorial at bytecode address `addr` with 1 argument
```

#### Two-Pass Compilation

The compiler uses **two passes** to handle forward references:

1. **Pass 1 — Register functions**: Scan all function definitions and record their names and parameter counts. No bytecode is emitted yet.
2. **Pass 2 — Compile bodies**: Now compile each function's body. When we encounter a call to a function whose address isn't known yet (forward reference), we emit a placeholder and add it to a **patch list**.
3. **Post-pass — Patch**: After all functions are compiled, resolve all forward references by filling in the actual addresses.

This is what enables **mutual recursion** — `is_even()` can call `is_odd()` even if `is_odd()` is defined later in the source.

### Example: Generated Bytecode

For our factorial program, the compiler produces:

```
Address  Instruction         Comment
──────────────────────────────────────────────────────
0000     CALL addr=32 0      Preamble: call main() with 0 args
0003     POP                 Discard main's return value
0004     HALT                End program

─── factorial(int n) ─────────────────────────────────
0005     ENTER 0             No extra locals (just param n at slot 0)
0007     LOAD 0              Push n
0009     CONST 1             Push 1
0011     LTE                 n <= 1 ?
0012     JZ 17               If false, skip to line 17
0014     CONST 1             Push 1 (return value)
0016     RET                 Return 1

0017     LOAD 0              Push n
0019     LOAD 0              Push n    (for the multiplication)
0021     CONST 1             Push 1
0023     SUB                 n - 1
0024     CALL addr=5 1       ← RECURSIVE CALL to factorial(n-1)
0027     MUL                 n * factorial(n-1)
0028     RET                 Return the result

0029     CONST 0             Safety net (unreachable)
0031     RET

─── main() ───────────────────────────────────────────
0032     ENTER 1             1 local variable (x at slot 0)
0034     SCAN                Read integer from stdin
0035     STORE 0             x = scan()
0037     LOAD 0              Push x
0039     CALL addr=5 1       Call factorial(x)
0042     PRINT               Print the result
0043     POP                 Discard print's dummy return value
0044     CONST 0             Push 0
0046     RET                 Return 0

0047     CONST 0             Safety net
0049     RET
```

### Key Design Decisions

- **Stack-based bytecode** (vs register-based): simpler to implement, easier to understand, and naturally maps to expression evaluation with a stack
- **Flat int array** for bytecode: opcodes and operands are interleaved in one array. Simple, cache-friendly, and avoids complex instruction encoding
- **Frame-relative addressing**: variables use `OP_LOAD slot` / `OP_STORE slot` where the slot is relative to the current frame pointer. This is what makes recursion work — each call gets a different fp
- **Implicit return**: every function ends with `CONST 0; RET` as a safety net, so even if the programmer forgets `return`, the function won't crash

## Phase 4 — Execution (Virtual Machine)

**File:** `src/vm.c` · `src/vm.h`

### What It Does

The VM interprets the bytecode produced by the compiler. It maintains a **data stack** for values and a **call frame stack** for function calls, then executes instructions in a loop.

### VM State

```
┌── VM State ─────────────────────────────────────────────┐
│                                                         │
│  code[]     The bytecode array (read-only)              │
│  ip         Instruction Pointer — next instruction      │
│                                                         │
│  stack[]    Data stack for values                        │
│  sp         Stack Pointer — top of the value stack       │
│  fp         Frame Pointer — base of current frame       │
│                                                         │
│  frames[]   Call frame stack (return addresses + fps)    │
│  fc         Frame Count — depth of call stack            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### The Execution Loop

The VM runs a simple **fetch-decode-execute** loop:

```
while (true) {
    1. FETCH:   Read the opcode at code[ip], advance ip
    2. DECODE:  Read any operands (advance ip for each)
    3. EXECUTE: Perform the operation (push/pop stack, jump, etc.)
}
```

### Stack Frame Layout

When `OP_CALL` is executed, a new **stack frame** is created. This is the crucial mechanism that enables recursion.

```
                    Stack Layout During factorial(3)
    ┌──────────────────────────────────────────────────────┐
    │                                                      │
    │  ┌────────────────────────┐                          │
    │  │ main() frame           │  fp_main ──────┐         │
    │  │  slot 0: x = 3        │               │         │
    │  │  (pushed arg: 3)      │               │         │
    │  ├────────────────────────┤               │         │
    │  │ factorial(3) frame     │  fp_fact3 ────┼──┐      │
    │  │  slot 0: n = 3        │               │  │      │
    │  │  (pushed arg: 2)      │               │  │      │
    │  ├────────────────────────┤               │  │      │
    │  │ factorial(2) frame     │  fp_fact2 ────┼──┼──┐   │
    │  │  slot 0: n = 2        │               │  │  │   │
    │  │  (pushed arg: 1)      │               │  │  │   │
    │  ├────────────────────────┤               │  │  │   │
    │  │ factorial(1) frame     │  fp_fact1 ────┼──┼──┼─┐ │
    │  │  slot 0: n = 1        │               │  │  │ │ │
    │  └────────────────────────┘               │  │  │ │ │
    │                                           ▼  ▼  ▼ ▼ │
    │  Call Frame Stack:                                   │
    │    frames[0] = { return_ip for main,  fp_main  }     │
    │    frames[1] = { return_ip for fact3, fp_fact3 }     │
    │    frames[2] = { return_ip for fact2, fp_fact2 }     │
    │                                                      │
    └──────────────────────────────────────────────────────┘
```

Each frame has its **own copy of `n`** at its own `fp + 0`. When `factorial(1)` reads `n`, it reads `stack[fp_fact1 + 0]` which is `1`. When `factorial(3)` reads `n`, it reads `stack[fp_fact3 + 0]` which is `3`. They never interfere.

### How OP_CALL Works (Step by Step)

When executing `OP_CALL addr nargs`:

```
1. The caller has already pushed `nargs` argument values onto the stack

2. Save current context:
     frames[fc].return_ip = ip    (save where to resume after return)
     frames[fc].fp = fp           (save current frame pointer)
     fc++                         (increment frame count)

3. Set up new frame:
     fp = sp - nargs              (frame pointer → first argument)
     ip = addr                    (jump to function entry)
```

### How OP_RET Works (Step by Step)

When executing `OP_RET`:

```
1. Pop the return value from the stack:
     retval = stack[--sp]

2. Discard entire frame (args + locals):
     sp = fp                      (reset stack to before args)

3. Restore caller's context:
     fc--
     ip = frames[fc].return_ip    (jump back to caller)
     fp = frames[fc].fp           (restore caller's frame pointer)

4. Push return value for the caller:
     stack[sp++] = retval
```

### Key Design Decisions

- **Separate call frame stack**: rather than storing return addresses on the data stack (which is error-prone), a dedicated `frames[]` array keeps control flow information cleanly separated from data
- **Division-by-zero protection**: `OP_DIV` and `OP_MOD` check for zero divisors and report an error
- **Stack bounds checking**: every iteration checks for stack overflow/underflow to prevent crashes
- **`OP_PRINT` pushes 0**: after printing, a dummy value is pushed so that `print()` can be used in expression context if needed (consistency with function call semantics)

## How Recursion Works — A Complete Walkthrough

Let's trace `factorial(3)` through the entire VM execution to see exactly how recursion works:

### Initial State

```
main() calls factorial(3)
```

### Step 1: main() starts

```
ip=32  ENTER 1      → allocate 1 local slot (x)
ip=34  SCAN         → user types 3, stack: [3]
ip=35  STORE 0      → x = 3, stack: []
ip=37  LOAD 0       → push x=3, stack: [3]
ip=39  CALL 5, 1    → call factorial with 1 arg
                       Save frame: frames[0] = {ip=42, fp=0}
                       New: fp=sp-1=stack_index_of_3, ip=5
```

### Step 2: factorial(3) — n=3, not base case

```
ip=5   ENTER 0      → no extra locals
ip=7   LOAD 0       → push n=3, stack: [..., 3]
ip=9   CONST 1      → stack: [..., 3, 1]
ip=11  LTE          → 3<=1 = 0, stack: [..., 0]
ip=12  JZ 17        → 0 is zero, JUMP to 17 (skip base case)
ip=17  LOAD 0       → push n=3
ip=19  LOAD 0       → push n=3
ip=21  CONST 1      → push 1
ip=23  SUB          → 3-1=2
ip=24  CALL 5, 1    → call factorial(2)
                       Save frame, new fp, ip=5
```

### Step 3: factorial(2) — n=2, not base case

```
Same as above but n=2
→ calls factorial(1)
```

### Step 4: factorial(1) — n=1, BASE CASE!

```
ip=5   ENTER 0
ip=7   LOAD 0       → push n=1
ip=9   CONST 1      → push 1
ip=11  LTE          → 1<=1 = 1 (TRUE!)
ip=12  JZ 17        → 1 is NOT zero, DON'T jump
ip=14  CONST 1      → push return value 1
ip=16  RET          → return 1 to factorial(2)
```

### Step 5: Unwinding — factorial(2) resumes

```
ip=27  MUL          → n * factorial(n-1) = 2 * 1 = 2
ip=28  RET          → return 2 to factorial(3)
```

### Step 6: Unwinding — factorial(3) resumes

```
ip=27  MUL          → n * factorial(n-1) = 3 * 2 = 6
ip=28  RET          → return 6 to main()
```

### Step 7: main() resumes

```
ip=42  PRINT        → print 6, output: "6"
ip=43  POP          → discard dummy
ip=44  CONST 0
ip=46  RET          → return 0
→ HALT
```

**Output: `6`** ✅ (3! = 6)

reccompiler edit myprogram.c

## Example Programs

### Factorial (Basic Recursion)

```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}

int main() {
    int n = scan();
    print(factorial(n));
    return 0;
}
```

### Fibonacci (Tree Recursion)

Two recursive calls per invocation — demonstrates branching recursion:

```c
int fib(int n) {
    if (n <= 0) return 0;
    if (n == 1) return 1;
    return fib(n - 1) + fib(n - 2);
}

int main() {
    int count = scan();
    int i = 0;
    for (i = 0; i < count; i = i + 1) {
        print(fib(i));
    }
    return 0;
}
```

### Mutual Recursion (Indirect Recursion)

Two functions calling each other — `is_even` calls `is_odd` and vice versa:

```c
int is_odd(int n) {
    if (n == 0) return 0;
    return is_even(n - 1);
}

int is_even(int n) {
    if (n == 0) return 1;
    return is_odd(n - 1);
}

int main() {
    int n = scan();
    int even = is_even(n);
    if (even) {
        print(1);
    } else {
        print(0);
    }
    return 0;
}
```

### GCD (Euclidean Algorithm)

```c
int gcd(int a, int b) {
    if (b == 0) return a;
    return gcd(b, a % b);
}

int main() {
    int a = scan();
    int b = scan();
    print(gcd(a, b));
    return 0;
}
```

### Power Function

```c
int power(int base, int exp) {
    if (exp == 0) return 1;
    return base * power(base, exp - 1);
}

int main() {
    int b = scan();
    int e = scan();
    print(power(b, e));
    return 0;
}
```
